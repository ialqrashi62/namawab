# نطاق النموذج الأولي للواجهات البرمجية - بيئة الاختبار (Staging API Prototype Scope)

* **المشروع:** منصة نما الطبية (NamaMedical ERP)
* **المرحلة:** جاهزية بيئة الاختبار وفجوات البيئة (PHASE_STAGING_READINESS_AND_ENVIRONMENT_GAP_REPORT)
* **الهدف:** تحديد ما هو مسموح وما هو ممنوع صراحة في التطوير القادم للنموذج البرمجي الأولي (API Prototype) لضمان العمل الآمن داخل بيئة الاختبار دون تعديل البيانات الأساسية.

---

## 1. العمليات المسموح بها في النموذج الأولي (Allowed Scope)

عند بناء النموذج الأولي اللاحق للواجهات البرمجية (API Prototype)، يُسمح فقط بالخصائص التالية:

* **واجهات القراءة فقط (GET/Read-Only Placeholder Endpoints):**
  * تقديم بيانات تجريبية ثابتة أو مستخرجة من ذاكرة خفيفة ومؤقتة.
  * مثال: عرض قائمة المرافق (`/api/v1/facilities`) وقائمة الأقسام (`/api/v1/departments`) بالاعتماد على مصفوفات ثابتة (Static Arrays) مسبقة الإعداد.
* **استجابات وهمية ثابتة (Static/Mock-Backed Responses):**
  * إرجاع كائنات استجابة مبنية على عقود البيانات DTO المحددة في ملف [enterprise-contracts.js](file:///c:/Users/ice/Desktop/NamaMedical/namaweb/public/js/enterprise-contracts.js).
  * لا يتم إجراء أي استعلامات معقدة أو تعديلات على الجداول الرئيسية.
* **فحص صلاحيات بسيط (Auth Guard Placeholders):**
  * التحقق من دور المستخدم (مثل طبيب، صيدلي، محاسب) عبر جلسة العمل الحالية وعرض رسالة خطأ أو نجاح تجريبية، دون دمج معقد أو تعديل في صلاحيات المستخدمين الفعلية.
* **معاينة سجلات التدقيق (Audit Preview Only):**
  * طباعة سجلات العمليات السريرية والمالية في وحدة التحكم (Console Logs) أو حفظها في مصفوفة مؤقتة في الذاكرة (In-Memory) لعرض كيف ستعمل أنظمة التدقيق، دون الكتابة في جداول التدقيق بقاعدة البيانات.

## 2. العمليات الممنوعة تماماً في النموذج الأولي (Prohibited Scope)

يُحظر تماماً وبشكل قاطع تضمين أي من العمليات التالية في المرحلة القادمة:

* **عمليات الكتابة الحقيقية (POST/PUT/DELETE Real Writes):**
  * يُمنع إنشاء أو تعديل أو حذف أي سجلات في جداول المرضى، الحجوزات، الوصفات الطبية، الفواتير، أو القيود المحاسبية.
* **إرسال المطالبات المالية للتأمين (Billing Submission):**
  * يُمنع منعاً باتاً محاولة إرسال أي فاتورة أو مطالبة مالية أو محاولة معالجتها ضد شركات التأمين.
* **الصرف الفعلي للأدوية (Pharmacy Dispense):**
  * يُمنع تعديل مخزون الأدوية الفعلي أو تسجيل صرف الدواء في سجلات المرضى.
* **التوقيع النهائي على السجلات الطبية (Final Signature):**
  * يُمنع إقفال الملف الطبي للمريض (SOAP Notes Lock) أو إجراء أي توقيع إلكتروني نهائي يؤدي لحبس السجل.
* **الاتصال الفعلي بنظام NPHIES (Real NPHIES Calls):**
  * يُمنع إرسال أي طلبات شبكية حقيقية إلى خوادم منصة نفيس (NPHIES) الحكومية.
* **الاتصال بقاعدة بيانات الإنتاج (Production DB Connection):**
  * يُمنع إدراج أو استخدام أي بيانات اتصال تشير إلى قاعدة بيانات الإنتاج الفعلي.

## 3. مصفوفة نطاق عمل الواجهات (API Scope Matrix)

| الواجهة البرمجية (Endpoint) | النطاق المسموح به في Staging | النطاق المحظور | طريقة المحاكاة |
| :--- | :--- | :--- | :--- |
| `/api/v1/facilities` | استرجاع المرافق الثابتة | تعديل أو إضافة مرفق | قراءة من مصفوفة ثابتة في الكود. |
| `/api/v1/departments` | استرجاع الأقسام وتصفيتها | تعديل الأقسام | تصفية محلية في الذاكرة. |
| `/api/v1/queue` | عرض دور المرضى الافتراضي | تعديل الحالة أو استدعاء مريض | عرض مصفوفة ثابتة وتحديثها في الذاكرة فقط. |
| `/api/v1/encounters` | عرض اللقاءات الطبية التجريبية | إنشاء لقاء جديد أو إقفاله | بيانات وهمية (Mock Data) مطابقة للـ DTO. |
| `/api/v1/clinical-orders` | عرض الطلبات الطبية الافتراضية | إرسال طلب جديد للمختبر/الأشعة | عرض ثابت ومحاكاة التحقق من الصلاحيات. |
| `/api/v1/clinical-results` | عرض نتائج افتراضية | إعطاء نتائج حقيقية للمرضى | بيانات ثابتة محددة مسبقاً. |
| `/api/v1/clinical-pharmacy`| عرض الأدوية المطلوبة للمراجعة | الصرف الفعلي للدواء | محاكاة فحص التداخل الدوائي في الذاكرة فقط. |
| `/api/v1/clinical-billing` | عرض معاينة الحساب الافتراضية | الدفع الفعلي أو إرسال التأمين | حساب نسبي افتراضي بالاعتماد على مصفوفة أسعار وهمية. |

---
**الخلاصة:** سيلتزم النموذج البرمجي الأولي القادم بالحدود الصارمة للقراءة فقط والمحاكاة الكاملة (Mocking)، بما يضمن بيئة اختبار آمنة وخالية من المخاطر.
