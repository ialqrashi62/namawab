# تقرير حدود الأمان لبيئة الاختبار - منصة نما الطبية (Staging Security Boundary Report)

* **المشروع:** منصة نما الطبية (NamaMedical ERP)
* **المرحلة:** جاهزية بيئة الاختبار وفجوات البيئة (PHASE_STAGING_READINESS_AND_ENVIRONMENT_GAP_REPORT)
* **الهدف:** توثيق الحدود الأمنية الصارمة وكيفية منع تداخل البيانات بين بيئات التطوير، الاختبار (Staging)، والإنتاج (Production) للامتثال للمعايير الوطنية السعودية لحماية البيانات الصحية والمالية.

---

## 1. حواجز الأمان على مستوى الشفرة البرمجية (Code-Level Boundaries)

تم تأسيس حواجز أمان برمجية ذاتية داخل كود التطبيق تمنع تفعيل أي ميزات حية بشكل تلقائي:

* **ضوابط العقود الذاتية (Self-Enforcing Contracts):**
  * في الملف [enterprise-contracts.js](file:///c:/Users/ice/Desktop/NamaMedical/namaweb/public/js/enterprise-contracts.js)، تم تثبيت الدوال:
    * `window.isLiveEndpointEnabled() { return false; }`
    * `window.isWriteOperationEnabled() { return false; }`
  * هذه الدوال تعمل كصمامات أمان مركزية (Kill Switches)؛ حتى في حال محاولة استدعاء الواجهات، يتم رفض الطلب فوراً على مستوى الواجهة الأمامية والخلفية إذا كانت هذه الأعلام `false`.
* **تعطيل مسارات واجهات التطبيق (OpenAPI Route Guards):**
  * جميع مسارات الواجهات البرمجية للمؤسسة مصنفة كـ `PLANNED_DISABLED`. لا توجد مسارات نشطة تؤدي لتحديث جداول قاعدة البيانات الحية.

## 2. حدود عزل الشبكة وقاعدة البيانات (Network & Database Boundaries)

لضمان العزل المادي التام، يجب تطبيق السياسات الأمنية التالية عند إعداد خادم Staging:

```mermaid
graph TD
    subgraph بيئة الإنتاج الفعلي (Production Site)
        ProdApp[تطبيق الإنتاج PM2] -->|اتصال مشفر| ProdDB[(قاعدة بيانات الإنتاج)]
    end

    subgraph بيئة الاختبار المعزولة (Staging Site)
        StgApp[تطبيق الاختبار PM2] -->|اتصال مشفر| StgDB[(قاعدة بيانات الاختبار)]
    end

    ProdApp -.->|جدار حماية يحظر الاتصال| StgDB
    StgApp -.->|جدار حماية يحظر الاتصال| ProdDB
    StgApp -.->|محاكاة حظر الارتباط| NPHIES[خوادم نفيس الحكومية]
```

* **عزل الشبكة (Network Isolation):**
  * يجب إعداد جدار حماية (Firewall) يمنع خادم Staging تماماً من الوصول إلى المنفذ `5432` الخاص بخادم الإنتاج.
* **عزل قاعدة البيانات (Database Isolation):**
  * يجب استخدام قاعدة بيانات منفصلة تماماً ومستقلة جغرافياً أو هيكلياً.
  * يمنع منعاً باتاً استخدام مستخدم قاعدة البيانات الخاص بالإنتاج في ملف `.env` الخاص بالـ Staging.

## 3. الامتثال للأنظمة التشريعية السعودية (Saudi Regulatory Compliance)

يتوافق تصميم حدود الأمان مع المتطلبات الوطنية التالية:

* **نظام حماية البيانات الشخصية (PDPL):**
  * يُحظر نقل أي بيانات حقيقية للمرضى من خادم الإنتاج إلى خادم Staging لأغراض الفحص أو التطوير.
  * يجب استخدام بيانات مصطنعة أو وهمية (Mock Data/Anonymized Data) بالكامل لضمان عدم الكشف عن الهوية السريرية لأي مريض.
* **متطلبات هيئة كفالة جودة الرعاية الصحية (CBAHI):**
  * حماية سجلات التدقيق (Audit Logs). في بيئة الاختبار، يتم توجيه سجلات التدقيق إلى ملفات محلية معزولة أو إلى الذاكرة فقط، لضمان عدم تلويث سجلات التدقيق الرسمية المعتمدة للإنتاج.
* **متطلبات هيئة الزكاة والضريبة والجمارك (ZATCA Phase 2):**
  * لمنع إرسال فواتير تجريبية إلى منصة الزكاة الحية، يتم تعطيل وحدة الربط الإلكتروني (ZATCA Integration) بالكامل في بيئة Staging أو تشغيلها في وضع المحاكاة الجافة (Dry-Run Mode) المعتمد من الهيئة للاختبار.
* **متطلبات منصة نفيس الوطنية (NPHIES):**
  * يتم حظر إرسال أي معاملات صحية أو مالية حقيقية إلى خوادم نفيس. جميع استجابات التأمين والمطالبات في الـ Staging تكون وهمية بالكامل بناءً على عقود البيانات المحددة.

## 4. إدارة الأسرار والتشفير (Secrets & Encryption)

* **عزل المفاتيح (Secrets Isolation):**
  * يجب استخدام مفتاح جلسة مستقل `SESSION_SECRET` ومفتاح توقيع مستقل `JWT_SECRET` لبيئة Staging.
  * استخدام نفس المفاتيح في البيئتين يعتبر خرقاً أمنياً من الدرجة الأولى ويؤدي لحجب البيئة فوراً.
* **تشفير البيانات الحساسة:**
  * يجب تفعيل التشفير في قاعدة بيانات Staging على مستوى الملفات والمحركات للتأكد من محاكاة البيئة الآمنة للإنتاج بشكل صحيح دون تسريب أي بيانات تجريبية.
